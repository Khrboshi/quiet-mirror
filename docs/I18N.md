# Internationalisation (i18n) in Quiet Mirror

Quiet Mirror ships in **6 languages** using a **custom TypeScript-based i18n system**. This doc describes the architecture, the rules every contributor is expected to follow, and how to extend the system.

If you are trying to fix a hardcoded English string, go straight to ["The metadata rule"](#the-metadata-rule) and ["Adding a translation"](#adding-a-translation).

---

## Supported languages

Quiet Mirror currently ships in **six languages**, including one right-to-left script (Arabic).

**The authoritative list lives in `app/lib/i18n/locales.ts`** — specifically `LOCALE_REGISTRY`, which carries each locale's code, display label, flag, text direction, and AI language name. Treat that file as the single source of truth; this doc deliberately does not enumerate the codes to avoid drift.

English is the default. The AI language name in the registry is used by `app/lib/ai/generateReflection.ts` to instruct the model which language to reply in.

There are **no plans to add more languages** unless explicitly requested. Quality over breadth — every locale must be 100% structurally complete (enforced at compile time) and human-checked for tone.

---

## Architecture

The system is deliberately small and library-free. No `next-intl`, no JSON catalogues, no runtime fetching.

### Files

```
app/lib/i18n/
├── index.ts        — barrel export (import from here)
├── types.ts        — the Translations interface (~1000 lines)
├── locales.ts      — the LOCALE_REGISTRY + helpers
├── server.ts       — getRequestLocale / getRequestTranslations
├── en.ts           — English strings (canonical source)
├── uk.ts           — Ukrainian
├── ar.ts           — Arabic
├── fr.ts           — French
├── nl.ts           — Dutch
└── ro.ts           — Romanian
```

### Type safety

`types.ts` defines the `Translations` interface. Every locale file exports a `const` of type `Translations` — so if you add a key to `en.ts` without adding it to the other 5, `tsc --noEmit` fails the build. This is the primary correctness mechanism; no runtime check is needed.

Function-valued entries like `(appName: string) => string` are used whenever a translation must splice in a runtime value (brand name, user name, a count). Never concatenate strings with `+` or template literals at the call site — put the interpolation inside the translation function so translators can reorder tokens for their grammar.

### Locale resolution

On the server (layouts, pages, route handlers):

```ts
import { getRequestTranslations } from "@/app/lib/i18n/server";

const t = await getRequestTranslations();
```

`getRequestTranslations()` reads the `qm:locale` cookie from the current request, validates it against `SUPPORTED_LOCALES`, and returns a fully-typed `Translations` object. Falls back to `DEFAULT_LOCALE` on anything invalid.

On the client:

```ts
import { useTranslation } from "@/app/components/I18nProvider";

const { t } = useTranslation();
```

`I18nProvider` hydrates from the same cookie on first render and exposes `setLocale()` for the language switcher.

### RTL

Handled automatically. The root `app/layout.tsx` sets `<html lang={locale} dir={dir}>` based on `getDir(locale)`, so Arabic flips layout for every page with zero per-component work.

## The metadata rule

**Never use `export const metadata` in a page or layout that ships user-facing text.** Static metadata is frozen at build time — the English strings will show up in non-English users' browser tabs, Open Graph cards, and search results, regardless of their locale cookie.

Use `generateMetadata` instead:

```ts
// app/install/layout.tsx — the reference example
import type { Metadata } from "next";
import { CONFIG } from "@/app/lib/config";
import { getRequestTranslations } from "@/app/lib/i18n/server";

export async function generateMetadata(): Promise<Metadata> {
  const t = await getRequestTranslations();
  return {
    title: t.installPage.metaTitle(CONFIG.appName),
    description: t.installPage.metaDescription(CONFIG.appName),
  };
}
```

Three rules inside that function:

1. **Brand name comes from `CONFIG.appName`** (`app/lib/config.ts`) — never hardcoded. Rebranding is a one-file change.
2. **Text strings come from `t.*`** — never hardcoded, even in English. English lives in `en.ts` alongside every other locale.
3. **Non-locale-sensitive metadata** (`robots`, `icons`, `manifest`, `keywords`, `appleWebApp`, etc.) stays as literals. That's fine — they don't vary by language.

### Enforced by ESLint

`.eslintrc.json` bans `export const metadata` in `app/**` via `no-restricted-syntax`. Allowlist for legitimate exceptions:

- `app/privacy/page.tsx`, `app/terms/page.tsx` — legal pages are intentionally English, paired with `LegalLanguageNotice`.
- `app/blog/[slug]/page.tsx` — article titles are content, not UI.
- `app/layout.tsx` — root only uses `CONFIG.*` / `BRAND.*` references; no raw locale-sensitive literals.
- `app/(protected)/layout.tsx` — only sets `robots`, no text.
- `app/api/**` — route handlers have no metadata exports.

If you're adding metadata that genuinely shouldn't vary by locale (e.g., a page that only sets `robots`), use:

```ts
// eslint-disable-next-line no-restricted-syntax -- justification goes here
export const metadata = { robots: { index: false, follow: false } };
```

Always write the justification; empty disables get rejected.

## Adding a translation

Most common task: you're adding a new UI string or a new page.

1. **Add the key to `types.ts`** under the appropriate block. If the string interpolates the brand name or any runtime value, type it as a function:
   ```ts
   // in Translations interface
   installPage: {
     metaTitle:       (appName: string) => string;
     metaDescription: (appName: string) => string;
     // ...
   }
   ```
2. **Populate `en.ts` first** — this is the canonical source. Match the existing indentation and quote style in the block you're editing.
3. **Populate every other locale file** (see `LOCALE_REGISTRY` in `locales.ts` for the current list). Non-English files use single quotes with backticks for template literals; English uses double quotes with column-aligned keys. Match each file's existing style.
4. **Run `npx tsc --noEmit`** — if you missed a locale or a key, the build fails with a precise error.
5. **Use it at the call site**:
   ```ts
   const t = await getRequestTranslations();
   const title = t.installPage.metaTitle(CONFIG.appName);
   ```

### Tips for non-English translations

- **Never translate the brand name.** `CONFIG.appName` always renders as "Quiet Mirror", in every locale.
- **Never translate user journal content.** The AI reflects back in the entry's language, not the UI's.
- Use `Intl.DateTimeFormat` / `Intl.NumberFormat` with `getIntlLocale(code)` for dates and numbers — not hand-formatted strings.
- For grammar-sensitive strings (pluralisation, gendered forms), use a function entry and handle the variants inside the translation, not at the call site.

## Adding a new locale

This is rare — the six-language scope is deliberately fixed. When it does happen, **follow the step-by-step instructions at the top of `app/lib/i18n/locales.ts`**, which is the canonical reference for the mechanics. That file's header comment stays in sync with the code; this doc intentionally does not duplicate it.

Removing a locale is symmetric and equally self-healing — also documented there.

## Known debt

The original six layouts/pages flagged for migration (#87–#92) have all been resolved in PRs #95–#100. The metadata rule (see [The metadata rule](#the-metadata-rule)) now catches any new `export const metadata` on sight.

One remaining file is covered by an explicit ESLint exclusion in `.eslintrc.json` because its content is structurally English (legal text with a translated notice banner), but its **title and description should still localise** so browser tabs and social previews match the user's language:

- `app/terms/page.tsx` — tracked for the next PR in this sequence

After that, the only remaining excluded files are intentional and have no locale-sensitive text:

- `app/layout.tsx` — root brand metadata (static by convention)
- `app/(protected)/layout.tsx` — `robots: noindex` only, no translatable strings
- `app/blog/[slug]/page.tsx` — per-post metadata derived from MDX frontmatter
- `app/api/**` — not user-facing routes

Each migration follows the same recipe: add `metaTitle` / `metaDescription` (and OG variants where the file uses them) to `legalPages` or the page's own namespace in all 6 locales, replace `export const metadata` with `generateMetadata` reading from `getRequestTranslations()`, and remove the file from the ESLint `excludedFiles` list so future regressions are caught.
